**文件**：`client/voice/tts_engine.py`

- 单条播报最长 120 秒超时保护
- 实机测试：三条连续文本均正常播报

**文件**：`client/audit/audit_agent.py`

### 8. OmniParser GPU 环境搭建 ✅

**相关文档**：`项目文档/OmniParser GPU 环境交接文档.md`

在学校分配的 **NVIDIA A800-SXM4-80GB**（80GB 显存）独立 GPU 容器中完成了微软开源 OmniParser v2.0 的完整环境部署与调试，为后续 SoM（Set-of-Mark）视觉标注能力提供本地化 AI 推理基础。

**环境配置**：
- Python 3.10.12 + PyTorch 2.7.1+cu118（**必须 cu118**，驱动 535.309.01 最高支持 CUDA 12.2，cu121/cu130 不可用）
- Transformers 4.43.4（**必须降级**，新版不兼容 Florence-2 模型）
- PaddleOCR 2.8.1 + PaddlePaddle-GPU 2.6.1（**必须锁定版本**，3.x 移除了 `use_dilation` 等旧参数）
- Gradio 4.44.1 + 国内镜像源（上交 SJTU PyTorch 镜像 + HuggingFace hf-mirror.com）
- 模型权重：YOLO 检测模型（OmniParser-v2.0）+ Florence-2-large 视觉语言模型（A800 显存充足，使用 Large 版）

**解决的 4 个关键兼容性问题**：

| # | 问题 | 根因 | 解决方案 |
|---|------|------|----------|
| 1 | `flash_attn` 编译失败 | 容器缺少 `nvcc` 编译器，无法编译 C++ 源码 | 修改 `transformers/dynamic_module_utils.py`，注释 `raise ImportError`，强制跳过依赖检查 |
| 2 | Florence-2 内部调用 `flash_attn` | 模型缓存代码尝试 import `flash_attn` | 修改 `.cache/huggingface/.../modeling_florence2.py`，强制 `is_flash_attn_2_available()` 返回 `False`，注释 import 语句 |
| 3 | Gradio JSON Schema 解析崩溃 | Gradio 4.44.1 对 `additionalProperties: false` 未做类型判断 | 修改 `gradio_client/utils.py` 第 863 行，增加 `isinstance(schema, dict)` 判断 |
| 4 | OmniParser OCR 空值/类型异常 | 无文字截图时未判空 + 排序 lambda 不兼容字典/列表混合 | 修改 `OmniParser/util/utils.py`，增加 `None` → `[]` 赋空 + `safe_get_content()` 兼容函数 |

**验证产出**：
- 编写 `test_complete.py` 测试脚本，输出 `result.json`（结构化元素数据）和 `output_labeled.png`（绿色 YOLO 标注框 + 红色 OCR 标注框）
- YOLO + Florence-2 + PaddleOCR 三模型联合推理链路调通

## 四、遗留问题

1. **Google Web Speech 需网络直连**，如遇网络受限需切换到百度语音 API（中文识别更优，免费 5 万次/天）
2. **pyttsx3 不支持指定输出设备**，始终使用 Windows 系统默认设备，需手动切换
3. **B-C 信号联调**（Day 3）依赖 B 的 PyQt5 GUI 完成
4. **OmniParser Gradio 界面**在学校内网代理下存在 POST 请求限制，建议通过 FastAPI 封装 RESTful 端点替代
5. **Florence-2-large 模型**为 HuggingFace 受限模型，新接手人需先在网页同意协议并 `hf auth login` 登录
6. **GPU 容器依赖严格锁定**：PyTorch cu118、PaddleOCR 2.8.x、Transformers 4.43.x 不可随意升级，详见交接文档
