以下是针对您三个问题的详细技术解析：

### 问题一：OmniParser、SoM 是如何与 LLM 结合起来的？（“感知→编号→推理→执行”闭环）

这三者的结合本质上是**将 LLM 不擅长的“连续空间坐标回归”问题，转化为其擅长的“离散空间语义选择”问题**。具体结合链路如下：

1. **感知层（OmniParser → 结构化数据）**：OmniParser 对屏幕截图进行推理，输出结构化的元素列表 \( E \)，包含每个 UI 元素的边界框（bbox）、类型（type）和文字（text）。
2. **标注层（SoM → 视觉提示 + 映射表）**：
   - **生成标注图**：系统在原始截图上，为每个元素绘制彩色边界框，并在其顶部叠加唯一的数字编号标签（如 `~1`, `~2`, `~3`），生成一张全新的**标注图（I_som）**。
   - **建立映射表**：同时维护一个内存字典 `Map{ "~3" : {bbox, center_coord, type} }`。
3. **推理层（多模态 LLM）**：将 **SoM 标注图（I_som）** 作为图像输入，将用户问题 + 格式约束（“请输出 JSON，引用 ~id”）作为文本输入，一起送入多模态大模型（GPT-4V/Qwen-VL）。
4. **解码与执行层（坐标反查）**：LLM 输出 `{"target_element_id": "~3"}`。系统截取该 ID，查询 `Map` 表，获得精确的屏幕像素坐标（center），最后交由 ANNO 模块在屏幕上渲染红色箭头和高亮框。

> **算法设计核心**：此设计将 LLM 输出像素坐标的准确率从 ~40% 暴力提升至 ~95%（引用自 SoM 论文数据），因为对于 LLM 来说，从 N 个标签中选出 `~3` 远比凭空生成 `(x, y)` 数值要稳定得多。

### 问题二：HAJIMI 项目和一般的 Chat LLM（如 ChatGPT）有什么区别？

HAJIMI 不是“通用对话机器人”，而是**“任务导向的具身智能体（Embodied Agent）”**，区别体现在五个维度：

| 维度 | **通用 Chat LLM** | **HAJIMI 智能指引助手** |
| :--- | :--- | :--- |
| **输入模态** | 纯文本（或单张通用图片） | **实时高分辨率屏幕上下文** + 用户语音/文本 |
| **输出形式** | 生成文字描述或代码 | **多模态执行**：屏幕覆盖层（箭头/高亮）+ 结构化步骤 + TTS 语音 |
| **知识边界** | 依赖训练数据中的常识（可能过时） | **实时感知当前桌面 UI**，绝不依赖记忆中的旧版软件界面 |
| **交互范式** | 单轮/多轮对话，无物理约束 | **有状态的任务蓝图** + 严格的物理安全红线（只指路，不操作） |
| **记忆机制** | 上下文窗口（如 128k tokens） | **分层记忆**：短期对话（滑动窗口）+ 长期蓝图（状态机独立存储）+ 屏幕指纹（环境状态验证） |

### 问题三：意图理解和蓝图规划怎么执行？算法设计点在哪？

这是 HAJIMI 的“决策中枢”，分为两大阶段，包含 4 个核心算法设计点。

#### 1. 意图理解（INTENT）—— 三层消歧算法
**执行流程**：
- **第一层（结构化解析）**：使用 `jieba` 词性标注提取“动词+名词”核心目标，调用微调后的 `BERT-base-chinese` 进行 9 大意图域分类（如操作指引、内容认知）。
- **第二层（指代消解）**：将用户的模糊指代（“这个”“左上角蓝色的”“圆圆的齿轮”）通过 5 种策略（显式文本匹配、空间推理、鼠标坐标映射、VLM语义匹配、上下文接力）映射为具体的 SoM 元素 ID。
- **第三层（置信度融合与主动澄清）**：计算综合置信度 \( \text{Conf} = 0.4P_{\text{intent}} + 0.4P_{\text{referent}} + 0.2P_{\text{context}} \)。**若 < 0.8**，触发主动澄清（生成二选一问题），将用户回答作为新锚点追加，**不重新截图**，直接修正意图。

#### 2. 蓝图规划（PLANNER）—— 状态机与指纹验证
**执行流程**：
- **L2/L3 路由**：通过复杂度评分 \( \text{score} = 0.3\cdot\text{len}(Q) + 8\cdot\#\text{verb} + 15\cdot\mathbb{1}_{\text{crossApp}} \)。`score < 30` 走 L2 快路径（本地规则，<3s）；`≥ 30` 走 L3 慢路径。
- **L3 蓝图生成**：将 SoM 图 + 意图送入多模态 LLM，生成**恒定步骤序列（Constant Steps）**，锁定保存为“蓝图”。
- **状态机执行**：进入 7 状态机（GENERATED → EXECUTING → SUSPENDED → COMPLETED）。每执行一步，通过**屏幕指纹（SHA256 + Jaccard相似度）** 比对预期状态。若匹配率 ≥ 80% 则推进；若 < 80% 则蓝图挂起，询问用户“跳过/回退/终止”。

---

#### 核心算法设计亮点（精华总结）

1. **“离散化”桥接（SoM ID）**：用统一的 `~N` 编号作为计算机视觉（CV）、大语言模型（LLM）和图形渲染（GUI）之间的**通用数据总线**，彻底解决多模态异构数据对齐难题。
2. **恒定蓝图（Constant Steps）**：针对复杂任务，**首轮规划后锁定步骤**，后续执行不受 LLM 随机性影响，保证长流程的确定性，这是区别于普通 Agent 反复重规划的核心壁垒。
3. **鲁棒的屏幕指纹（Jaccard 而非 Hash）**：不直接用 SHA256（极易因像素微小变动全盘改变），而是提取“窗口类型 + Top5 元素集合”字段计算 Jaccard 相似度，允许界面在字体、分辨率上有浮动，容错性极强。
4. **”旁路”审计**：指引执行和审计上报完全异步解耦，不阻塞主链路。审计数据用于系统监控与优化。

综上，HAJIMI 的算法体系本质上是**用确定性的工程状态机（蓝图/指纹）去约束非确定性的生成式 AI（LLM）**，在保证“天马行空”的理解力的同时，锁死“安全、稳定、可回退”的执行底线。
