# 参考方案优先级与实施路线图

> 本文档基于 HAJIMI 代码实际状态，对感知层、理解层、规划层、执行验证层的候选辅助方案进行优先级排序和工时评估。判断标准不是文档写了什么，而是项目现在**真正缺什么、改了能见效**。

---

## 优先级总览

| 优先级 | 方案 | 工时 | 类型 | 解决什么 |
|--------|------|------|------|---------|
| **P0 紧急** | LLM prompt 注入 OmniParser 输出 | 3-4h | 代码改动 | LLM 对界面元素完全盲选 |
| **P1 高** | SetFit 替换关键词意图分类 | 4-6h | 模型集成 | 意图分类是 10 行 if-else |
| **P2 中高** | GroundingDINO 级联补漏 | 8-12h | 模型集成 | 非标准控件/模糊描述定位 |
| **P3 中** | XState 建模参考 → 状态迁移补完 | 2-3h | 设计审查 | 现有状态机有遗漏 |
| **P4 中低** | Rasa NLU 实体抽取借鉴 | 3-4h | 代码改动 | 用户约束条件提取 |
| **P5 低** | LangChain P&E + Plan-and-Solve | 1-2h | 文档参考 | 理论支撑，零代码 |
| **P5 低** | WebShop 策略参考 | 1-2h | 文档参考 | 指纹阈值论证 |

---

## 逐项详解

### P0：LLM prompt 注入 OmniParser 输出（3-4h）

**为什么排第一**：当前 `server/services/llm_ai.py:322` 步骤-元素绑定是 `elements[i % len(elements)]`，纯机械循环。LLM 根本没见过元素列表。这是系统最大的功能缺陷——生成了正确的步骤，却指向了错误的控件。

**改动范围**：
- `server/services/llm_ai.py`：改造 `SYSTEM_PROMPT`，注入元素列表；让 LLM 输出 `target_element_id`
- `server/services/llm_ai.py:318-340`：根据 LLM 返回的 `target_element_id` 绑定元素，而非循环

**风险**：低。纯 prompt 改动，不引入新依赖。

---

### P1：SetFit 替换关键词意图分类（4-6h）

**为什么排第二**：当前 `classify_intent()` 是 10 行 if-else 关键词匹配，只覆盖 6 种情况，置信度是硬编码的。文档声称的"BERT 主力分类"并未实现。

**改动范围**：
- 安装 `setfit`（`pip install setfit`）
- 为 9 类意图各准备 8-16 条标注样本
- 训练脚本 + 模型保存/加载
- 替换 `classify_intent()` 调用

**风险**：中低。需要标注少量数据。模型文件 ~100MB。训练 5 分钟。

---

### P2：GroundingDINO 级联补漏（8-12h）

**为什么排第三**：当用户说"那个圆圆的像齿轮的东西"时，OmniParser 检测不到（它只识别标准 UI 控件）。GroundingDINO 能做开放词汇检测，但需要 GPU。

**改动范围**：
- 部署 GroundingDINO checkpoint
- 在 `omniparser_client.py` 或 `ui_parser.py` 中加级联逻辑：OmniParser 置信度 < 阈值 → 触发 GroundingDINO
- 元素合并去重逻辑

**风险**：中高。引入新模型依赖，需 GPU 推理。与 OmniParser 的输出格式需要适配。

---

### P3：XState 状态迁移审查（2-3h）

**为什么排第四**：当前 `blueprint.py` 的 7 状态机是手工实现的 if-else。设计文档说 7 个状态，但实际代码只处理了基本迁移路径，异常分支（如 `executing` → `terminated` 的中间清理）未覆盖。

**改动范围**：
- 对照 XState 的 statechart 思想审查 `blueprint.py` 的状态迁移表
- 补遗漏的 guard 条件和迁移路径（如超时 → suspended、外部取消 → terminated）
- 纯设计审查 + 几行代码补完

**风险**：低。不需要引入新库，纯逻辑审查。

---

### P4：实体抽取（Rasa NLU 参考）（3-4h）

**为什么排第五**：用户说"不要装在 C 盘"时，系统需要提取 `{约束: 安装路径 ≠ C:}`。当前完全没有实体抽取。但这需要和蓝图引擎联动才能发挥作用，单独做效果有限。

**改动范围**：
- 参考 Rasa DIETClassifier 思路，在 prompt 中加实体抽取指令（非引入 Rasa）
- LLM 返回结构增加 `constraints` 字段
- 蓝图执行时检查约束条件

**风险**：低-中。纯 prompt 增强，但和蓝图联动需要额外设计。

---

### P5：纯文档参考类（2-4h 总计）

- **LangChain Plan-and-Execute**：已有 Constant Steps 蓝图，思想一致。花 1h 读文档，看 `Plan` 对象的数据结构设计，确认 `Blueprint` schema 是否完整。
- **Plan-and-Solve**：理论支撑，0 代码成本。可写进设计文档。
- **WebShop**：读它处理"部分状态匹配"的工程策略，0 代码成本。

---

## 建议实施顺序

```
第 1 周: P0 (4h)  →  LLM 能看见元素，步骤绑定从随机变语义
第 1 周: P5 (2h)  →  文档参考，顺手做
第 2 周: P1 (6h)  →  意图分类从 if-else 升级到 SetFit 模型
第 3 周: P3 (3h)  →  状态机补完
第 3 周: P4 (4h)  →  实体抽取
第 4-5 周: P2 (12h) → GroundingDINO 级联，模糊描述定位
```

**总计约 31-35 工时**，其中 **P0 + P1 + P3 这 13-16h 是性价比最高的组合**——解决了 LLM 盲选 + 意图猜测 + 状态遗漏三个真实 bug，且不引入任何新模型依赖。

---

## 关键决策建议

1. **先 P0，后 P1**：P0 解决的是系统能否正确"指到"元素，是用户体验的底线。P1 提升的是意图理解精度，属于优化项。
2. **P2 放在最后**：GroundingDINO 效果好但引入 GPU 依赖和级联复杂度，建议在 P0/P1/P3 稳定后再集成。
3. **P4 可并入 P1**：实体抽取和意图分类本质都是 NLU 任务，可以共用 SetFit 或 LLM-based 方案，不必单独立项。
4. **P5 贯穿全程**：文档参考工作不需要独立排期，可在每周末尾顺手完成。
